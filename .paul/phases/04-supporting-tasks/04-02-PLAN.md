## AC-1: Checklist Covers Core Quality Dimensions
```gherkin
Given the planning-quality checklist
When used to assess a PLANNING.md
Then it checks: metadata present, problem statement clear, type-specific sections covered, design decisions documented, phases defined, skill loadout noted
```

## AC-2: Type-Aware Assessment
```gherkin
Given the checklist
When applied to different project types
Then it adapts expectations (utilities need less depth than applications)
And references config.md required sections as the authority
```

## AC-3: PAUL-Readiness Check
```gherkin
Given the checklist
When assessing for headless PAUL init readiness
Then it verifies enough detail exists for PAUL to derive milestones and phases without asking questions
```

<task type="auto">
  <name>Task 1: Create checklists/planning-quality.md</name>
  <files>checklists/planning-quality.md</files>
  <action>
    Create the planning quality checklist. Structure:

    1. **Header** — Purpose: quality gate for graduate/launch
    2. **Universal checks** (all types):
       - [ ] Type metadata present
       - [ ] Problem statement is specific (not vague)
       - [ ] Design decisions documented (at least 1)
       - [ ] Open questions listed (even if "none")
       - [ ] Next actions defined
    3. **Type-specific checks** — reference data/{type}/config.md for required sections:
       - [ ] All required sections (per config.md) have substantive content
       - [ ] Optional sections either covered or explicitly skipped
    4. **PAUL-readiness checks** (for /seed launch):
       - [ ] Phase breakdown exists with independently shippable milestones
       - [ ] Each phase has build/test/outcome defined
       - [ ] Tech stack decisions are resolved (not "TBD")
       - [ ] Enough detail that PAUL can derive structure without questions
    5. **Scoring guidance** — not a numeric score, but a pass/warn/fail:
       - **Pass:** All universal + all required type-specific
       - **Warn:** Missing optional sections or thin content in some areas
       - **Fail:** Missing required sections or no phase breakdown

    Keep it concise — this is a reference checklist, not a framework document.
  </action>

    3. Contains type-specific reference to config.md
    4. Contains PAUL-readiness section
    5. Contains pass/warn/fail guidance
    6. Under 80 lines
  </verify>
  <done>AC-1 through AC-3 satisfied</done>
